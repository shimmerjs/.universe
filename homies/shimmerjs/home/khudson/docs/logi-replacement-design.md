# logi options+ replacement -- feasibility report + phase plan (MX Master 4, direct BT)

status: proposed. produced by a prior-art workflow (93 agents; 3-skeptic refute-quorum on load-bearing claims: 26 verified, 2 refuted-and-corrected, 123 cited-unverified, 78 above the verify cap). citation labels: [N] = quorum-verified claim, [U N] = cited but not quorum-verified (source named, not independently confirmed). the "corrections from verification" section restates every load-bearing verification inline. companion to magicbus-design.md (the logiretch module seam); the six design-doc corrections at the bottom have been folded into magicbus-design.md in the same commit.

user steer (verbatim, 2026-07-14): "figure out if its plausible to build out all of the functionality of the logitech app (esp in terms of input functionality [dpi, etc]) purely via our own substrate (hid bus and go toolchain/app). end goal is no logioptions app or anything in the family, entirely self managed. battery pct should have specialized icon in bottom row of khud (same one with home/kb/clod/caffeine/etc)"


## Top-line answer

**Yes -- the entire Options+ family is replaceable by the khudson substrate (touchd logiretch module + host-side consumers), with a short list of accepted app-only losses, conditional on the logiretch-0 spike.** Every daily-driver capability is either device-side HID++ 2.0 (proven on this exact device by logiops/Solaar prior art [U84][U89]), macOS-native (pointer accel, scroll speed, thumbwheel pan [U93][U33]), or host-reimplementable inside TCC grants khudson already holds (Input Monitoring in touchd, Accessibility in the khudson bus -- zero new prompts [U96]). The access mechanism (userland hidapi opening the 0xFF43 vendor collection over direct BT on macOS) is double-sourced: Solaar PR #2730, merged 2025-01-01, tested MX Master 3S + MX Keys [14][U109], plus OpenLogi as an independent shipping macOS replacement [U101, unverified]. What nobody anywhere has done is **MX Master 4 + macOS + direct BLE specifically** -- all three public MX4 feature dumps are Bolt-receiver captures [5][U77][U87] -- so the design doc's "conditional-GO pending one-device probe" (magicbus-design.md:80) stands and must not be upgraded by this research.

The genuinely lost-with-Options+ set is small and mostly wanted gone: Flow (LAN protocol, unreimplementable at reasonable cost [U104]), AI Prompt Builder / cloud sync / SSO [U32], the Options+-rendered Actions Ring overlay (rebuildable as our own UI), and firmware DFU (fallback: temporary Options+ install per update [U71][U105]). Dropping Options+ also removes a root-owned auto-updater LaunchDaemon and a multi-agent install footprint [U25, unverified].

## Phase fit: what fits vs what's forced

The decided seam survives intact: **single 0xFF43 vendor node over direct BT, non-exclusive open, never seize, battery-first, command lane via broadcaster v2 phase 4** (magicbus-design.md:76-100). Nothing found invalidates it [U123]. Note the doc's phase list ends at 5; "phase 6" below is proposed new numbering, not existing doc text (verifier correction).

**Fits as written:**
- Phase 5 battery-first: 0x1004 V5 confirmed on-device, getStatus + retained line, exactly the phase-1 logiretch capability [5][6]. Push events arrive free on the same read loop.
- Phase 4 command lane: DPI, SmartShift(0x2111), hi-res(0x2121), thumbwheel(0x2150), static remaps(0x1B04), haptic-level/play(0x19B0), force threshold(0x19C0), host-switch(0x1814) are all one-shot id-correlated request/reply writes [7][15][U89]. Adopt ratbagd-style dirty-flag commit batching -- not for device wear but because BLE serializes writes at the connection-interval floor (~25-60ms/transaction), so UI bursts should coalesce [U40][U41].

**Phase 5 must add three small items (load-bearing, cheap):** [U123]
1. swId-filtered request correlator with ~4s deadline (needed for battery anyway; unsolicited events carry swId 0, replies echo ours -- use a nonzero swId with MSB set, drop foreign-swId replies silently) [U118].
2. Parse 0x1D4B WIRELESS_DEVICE_STATUS from the FIRST deployment, published as a retained active/reconfNeeded line -- if macOS keeps the IOHID node alive across BLE idle-disconnect (unknown), 0x1D4B is the only reconnect signal we get [U111][U116].
3. A third failure class: request-timeout/link-down on an OPEN handle. Today's backoff taxonomy (30s absent / 5min seized, daemon.go:14-58) keys only on open failures; a sleeping mouse times out requests without failing open. This class must not feed the 5min seized backoff and must not close the handle [U120][3].

**NEW phase 5b -- logi config re-assert (forced, small):** no 0x8100, no 0x1C00; Solaar's ground truth is "devices forget most changed settings when turned off" [U110], corroborated by issues #2465/#2636/#686 and logiops #148/#66 [8][U64][U121]. Desired state lives in the CUE config channel (magicbus-design.md:63-68), replayed (a) unconditionally on every device (re)open -- exact local precedent: the Edge digitizer asserts mode on EVERY open, touchd/hid.go:119-122 (corrected line ref) -- and (b) on 0x1D4B reconfNeeded, cookie-deduped per Solaar's model (both triggers, belt-and-suspenders; notification-driven alone is proven insufficient) [U112][U113][U121]. Module-internal, not client commands.

**NEW phase 6a (daemon) + 6b (khudson consumer) -- divert engine:** gestures need (a) a notification router splitting unsolicited events from swId-matched replies on the single node, (b) diverted-button/rawXY event lines on logiretch.sock, (c) a host gesture state machine, (d) CGEvent synthesis. (c)+(d) must live outside touchd: event posting is Accessibility-gated, khudson already holds it (module.nix ~:36-44 -- doc's :49-50 ref is stale), touchd holds Input Monitoring only; adding Accessibility to the granted daemon violates TCC continuity [10][U96][U97]. Design-in mitigations: rawXY lines ride the evictable event queue (drop-oldest, broadcast.go:111-125; per-socket isolation doesn't protect logiretch's stream from itself) -- **coalesce dx/dy instead of dropping**, and discard the spurious first movement report on entering rawXY mode (Solaar #1662) [11].

**New consumers, zero touchd change:** per-app profile switcher (NSWorkspace watcher), actions/macros engine, Actions Ring overlay [15][U97].

## Transport rules for the Go module (source-verified against pinned go-hid v0.15.0 [17])

- `hid.Init()` BEFORE `SetOpenExclusive(false)`: pre-init the setter is silently clobbered and the first open would SEIZE the mouse (hid_init sets exclusive=1 for backward compat) [19]. Correction: touchd already calls hid.Init() deliberately at main.go:117 -- not "safe by accident" -- but the ordering rule must be stated for the logiretch module.
- `SetOpenExclusive(false)` maps exactly to kIOHIDOptionsTypeNone (non-seize); the flag is process-global, captured per-handle at open -- extend openMu serialization (hid.go:78-94) to logi opens [18][20][2]. Solaar mainline does the identical call on Darwin [22].
- Enumerate + filter VID 0x046D + UsagePage 0xFF43/Usage 0x0202; never `hid.Open(vid,pid)` [U8]. OpenPath opens the WHOLE IOHIDDevice: mouse reports interleave with 0x11 frames on one handle -- filter by report ID [25].
- SINGLE reader goroutine owns Read; per-handle queue is bounded drop-oldest at 32 reports (corrected from ~31) -- at ~125Hz motion a ~250ms reader stall starts evicting frames that may include a pending reply. Drain continuously; requests get timeouts + retry [23][24].
- Write HID++ via `Device.Write` (Output reports), never feature reports; macOS delivers BLE input reports with the leading 0x11 and full 20 bytes [21][U7][U49]. Read buffers >= 64 bytes [U6].
- Never cache DevSrvsID paths across disconnects; re-enumerate on every reopen (registry entry IDs are unique per service lifetime) [U47]. Removal callback -> read returns -1 maps onto the existing readLoop loss contract as-is [U46][U115].
- Numbered-write proof is a REAL spike gap: the moonlander write is unnumbered (moonlander.go:90-94) -- it proves the pipe, not HID++ framing [1].

## khud dock battery icon: cadence note (as requested)

**No protocol constraint threatens the bounded 60-300s cadence -- it holds with orders-of-magnitude slack.** 0x1004 is push-capable: unsolicited broadcasts at notification addr 0x00 arrive on the same vendor handle the module holds open, so the icon is event-fresh at zero marginal cost while the link is up; the bounded poll demotes to a reconcile floor for missed events after sleep/reconnect [4][U42][U62]. The BLE connection-interval floor (~25-60ms per transaction) is 3-4 orders below the poll budget [U40]. The real limit is availability, not rate: a napping mouse has no live link, so the icon must render **last-known-value + age**, never an error -- and the age display doubles as the staleness alarm for the OS-update regression class our backoff would otherwise mask [U103][U44]. Reattach latency after power-cycle is bounded by the 30s arrival scan (unless optional phase 3 ships) -- inside budget [U37]. Zero-HID++ fallback exists (ioreg BatteryPercent from cached GATT, no charging state; MX4 presence unverified) [U102]. Constant-cost invariant: O(1) per event, one timer, ~32-report bounded queue -- holds [24][U50].

## What we lose with Options+ gone, honestly

- **Lost, no replacement planned:** Flow (don't care -- Universal Control/Deskflow if ever), AI Prompt Builder/cloud/SSO (want gone), presentation-remote features. [U104][U32]
- **Lost, rebuildable when wanted:** Actions Ring overlay (our UI on diverted 0x01A0 + haptic tick), Options+-driven haptic events (the "feel Windows 11" integration is Windows-only -- nothing macOS-native is lost) [U30][U31], Smart Actions (our Go actions engine is strictly better) [U14].
- **Lost, workaround:** DFU -- temporary Options+ install per firmware update; worst case during the update our open fails into the 5min seized class and recovers [U71][U105].
- **Was never Options+'s to begin with:** pointer accel curve and tracking speed (macOS-native; Sonoma toggle) [U93], thumbwheel horizontal scroll (AC Pan) [U33], SmartShift auto-mode (device-side, last-written threshold) [U34]. Options+'s host-side magic was Accessibility-gated event synthesis for smooth scrolling/gestures [U91][U92] -- replaced by native feel or the 0x2121-divert path, which notably needs **no event tap anywhere** [U95].
- **Zero-software baseline** (nothing installed, our stack off): pointer + native accel, notchy line scroll, thumbwheel pan, middle click, SmartShift, battery % in System Settings (probably). Lost until our stack covers them: smooth scroll, gesture actions, per-app anything, ring, haptic events, remaps [U34, unverified].
- **Migration hazard:** Options+ may leave divert residue (1B04 flags, 2121 target) that outlives it -- the #3046 back-button inversion persisted until an explicit reset (correctable via reset + reboot; maintainer suspects MX4 retains divert state "maybe indefinitely" -- correction). First takeover must explicitly reset 1B04 remaps/diverts and 2121 target to defaults [13][U107]. Two HID++ masters config-fighting is a documented failure mode; uninstall Options+ first (the end goal anyway) [U119][U48].

## Risk register

**Spike-retirable (one session):** node topology/framing; device index; numbered writes; spontaneous notifications on macOS BLE; handle-death detection + re-enumeration; Options+ coexistence under shared open; feature-table absences; per-setting survival matrix [U122].

**Soak-class (time/rare-event):** IOHID node persistence across idle-disconnect variants; 0x1D4B missed-event rate across weeks; Options+ config-fights if it lingers during migration; the macOS-update BT regression class (bluez-5.77-analog: backoff masks it -- battery-age display is the alarm) [U44][U122]; write-while-link-down semantics.

**Design-level:** rawXY eviction mid-gesture (coalesce, don't drop) [11]; BLE write serialization (dirty-flag batching) [U40][U41]; hi-res wheel MX4 quirk [U83]; divert residue at takeover [U107].
